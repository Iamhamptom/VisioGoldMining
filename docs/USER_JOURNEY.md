# User Journeys — VisioGold DRC

This document describes three primary user journeys through the platform, referencing specific UI surfaces and code paths.

---

## Journey 1: Investor / Operator Evaluating a DRC Opportunity

**Persona**: A mining investor evaluating potential gold opportunities in eastern DRC.

### Step 1 — Login and Workspace Selection

The user navigates to the platform and logs in via the login page (`src/app/login/page.tsx`). The system authenticates via `POST /api/auth/login` (`src/app/api/auth/login/route.ts`), which validates credentials with bcrypt and issues a JWT containing the user's workspace context. The workspace switcher (`src/components/workspace-switcher.tsx`) allows selecting the active workspace.

### Step 2 — Explore the Globe

The user lands on the **Globe Home** screen (`src/components/screens/GlobeHome.tsx`), which renders an animated 3D globe via the `cobe` library. They click "Enter Map" to transition to the full **MapArea** (`src/components/MapArea.tsx`) with the DRC centered at coordinates [23.66, -2.88] (`src/lib/map-config.ts`).

### Step 3 — Toggle Layers

Using the **Layer Toggle** panel (`src/components/map/LayerToggle.tsx`), the user enables tenement boundaries (gold-colored polygons), mineral occurrences (colored circles by commodity), and security events (red/orange/yellow markers by severity). The layer definitions are registered in `src/lib/layers-registry.ts` and rendered by `src/components/map/GlobeMap.tsx`.

### Step 4 — Browse Opportunities

The **Opportunity Explorer** screen (`src/components/screens/OpportunityExplorer.tsx`) surfaces the **Opportunity Feed** (`src/components/opportunities/OpportunityFeed.tsx`) — a ranked list of scored tenements. Each **OpportunityCardItem** (`src/components/opportunities/OpportunityCardItem.tsx`) displays the composite score, five dimension scores (prospectivity, access, security, legal complexity, data completeness), and a natural-language explanation generated by the Opportunity Engine (`src/lib/opportunity-engine.ts`).

### Step 5 — Inspect a Specific Opportunity

Clicking an opportunity card zooms the map to the tenement and opens the **Feature Context Panel** (`src/components/panels/FeatureContextPanel.tsx`), which provides specialized cards:
- **LithologyCard** (`src/components/panels/LithologyCard.tsx`) — nearby geological formations
- **OccurrenceCard** (`src/components/panels/OccurrenceCard.tsx`) — mineral occurrences within range
- **IncidentCard** (`src/components/panels/IncidentCard.tsx`) — recent security events
- **InfrastructureCard** (`src/components/panels/InfrastructureCard.tsx`) — roads, airports, rivers
- **PermitCard** (`src/components/panels/PermitCard.tsx`) — permit details and status

### Step 6 — Create a Repo for Further Evaluation

If the opportunity warrants deeper evaluation, the user opens the **Repo Create Modal** (`src/components/repo-create-modal.tsx`) directly from the opportunity context. This creates a new Repo (via `POST /api/repos`) tied to the tenement's geographic polygon, with a default `main` branch.

### Step 7 — Run a Simulation

The user navigates to the **Simulation Studio** (`src/components/screens/SimulationStudio.tsx`), configures scenario parameters using **ScenarioSliders** (`src/components/simulation/ScenarioSliders.tsx`), and runs the simulation. The Express API (`server/src/routes/simulations.ts`) executes the cost engine, schedule engine, and risk engine. Results appear in the **CostBreakdownTable** (`src/components/simulation/CostBreakdownTable.tsx`), **ScheduleCard** (`src/components/simulation/ScheduleCard.tsx`), and **RiskScoreCards** (`src/components/simulation/RiskScoreCards.tsx`).

### Step 8 — Compare Scenarios

The user runs a second simulation with different parameters and uses **ScenarioCompare** (`src/components/simulation/ScenarioCompare.tsx`) to view per-department cost deltas, schedule differences, and risk changes side-by-side. The compare engine (`server/src/engine/compareEngine.ts`) generates a rules-based recommendation.

### Step 9 — Make a Decision

Based on the comparison, the user decides whether to proceed, adjusts their investment thesis, or archives the opportunity for future review.

## Journey 2: Analyst Building a New Repo and Attaching Evidence

**Persona**: A mining analyst conducting due diligence on a granted permit in Haut-Katanga.

### Step 1 — Create a Workspace (if needed)

If the analyst's organization doesn't have a workspace yet, an ADMIN or OWNER creates one via the Workspaces page (`src/app/(app)/workspaces/page.tsx`) using `POST /api/workspaces`. Team members are added via `POST /api/workspaces/:id/members`.

### Step 2 — Create a Repo

From the Repos page (`src/app/(app)/repos/page.tsx`), the analyst creates a new repo using the **Repo Create Modal** (`src/components/repo-create-modal.tsx`). They specify:
- Name (e.g., "Haut-Katanga PR-1188")
- Slug (e.g., "hk-pr-1188")
- Country: DRC
- Commodity: Gold
- Description: Brief context about the permit

The system creates the repo with a `main` branch via `POST /api/repos` and `POST /api/repos/:id/branches`.

### Step 3 — Upload Artifacts

On the branch page (`src/app/(app)/repos/[repoId]/branches/[branchId]/page.tsx`), the analyst uses the **Artifact Upload** component (`src/components/artifact-upload.tsx`) to attach files:
- Geological survey PDF (type: DOCUMENT)
- Soil sample dataset CSV (type: DATASET)
- Permit verification letter (type: DOCUMENT)
- Vendor cost estimate (type: VENDOR_REPORT)

Each upload is processed via `POST /api/branches/:id/artifacts`, which computes a SHA-256 hash (`src/lib/hash.ts`), stores the file (`src/lib/storage.ts`), and creates an immutable artifact record. The **Artifact Vault** (`src/components/artifact-vault.tsx`) displays all attached files with hashes and timestamps.

### Step 4 — Create a Commit

After uploading all evidence for this evaluation phase, the analyst creates a commit (via `POST /api/branches/:id/commits`) with a message like "Initial due diligence package — geology, permits, vendor quotes". The commit references all artifact changes and becomes an immutable snapshot. The **Commit Timeline** (`src/components/commit-timeline.tsx`) visualizes the branch history.

### Step 5 — Branch for Alternative Analysis

The analyst creates a new branch (via **Branch Tabs** — `src/components/branch-tabs.tsx`) called "conservative-estimate" to explore a more conservative evaluation path. They upload additional artifacts and create commits on this branch independently.

### Step 6 — Merge Findings

When the conservative analysis is complete, the analyst merges the branch back into `main` using `POST /api/commits/merge`. The system performs three-way merge with conflict detection. If conflicts arise, the system returns them for resolution (conflict resolution UI is planned — 🔴).

### Step 7 — Publish for Review

An ADMIN publishes a snapshot of the `main` branch using the **Publish Branch Dialog** (`src/components/publish-branch-dialog.tsx`), which calls `POST /api/branches/:id/publish`. They configure redaction rules to exclude internal vendor quotes. The snapshot becomes accessible at `/p/:slug` (`src/app/(public)/p/[slug]/page.tsx`).

### Step 8 — Audit Trail

The ADMIN reviews the complete audit trail via the Audit page (`src/app/(app)/audit/page.tsx`) and **Audit Log Viewer** (`src/components/audit-log-viewer.tsx`), which shows every action taken on the workspace with timestamps, user identities, and IP addresses.

## Journey 3: User Running Simulation and Generating a Project Plan

**Persona**: An operator who has identified a promising permit and wants to model the project economics and build a structured plan.

### Step 1 — Select the Repo

The user navigates to an existing repo via the Repos page and selects the `main` branch. The **RepoMapPanel** (`src/components/repos/RepoMapPanel.tsx`) shows the repo's geographic boundary on a mini-map.

### Step 2 — Open Simulation Studio

From the sidebar (`src/components/Sidebar.tsx`), the user navigates to the **Simulation Studio** (`src/components/screens/SimulationStudio.tsx`).

### Step 3 — Configure Scenario Parameters

Using the **ScenarioSliders** (`src/components/simulation/ScenarioSliders.tsx`), the user sets:
- **Project type**: Exploration / Small Mine / Industrial
- **Logistics mode**: Road / Mixed / Helicopter
- **Security posture**: Low / Medium / High
- **Camp standard**: Basic / Standard / Premium
- **Compliance rigor**: Minimum / Standard / Investor-grade
- **Drilling program**: Type (RC / Diamond / Mixed), meters, and mobilization
- **Sampling**: Sample count and assay package (Screening / Standard / Full QA/QC)
- **Lab location**: Local / Regional / International
- **Timeline aggressiveness**: Fast / Normal / Conservative
- **Gold price assumption**: USD per ounce

### Step 4 — Run Simulation

The user clicks "Run Simulation", which calls `POST /api/repos/:id/branches/:id/simulations/run` on the Express server. The simulation service (`server/src/services/simulation.service.ts`) orchestrates:

1. **Cost Engine** (`server/src/engine/costEngine.ts`) — 1,000 Monte Carlo iterations across 12 departments using DRC-calibrated priors (`server/src/config/priors.ts`), producing P10/P50/P90 cost distributions
2. **Schedule Engine** (`server/src/engine/scheduleEngine.ts`) — Phase-by-phase timeline estimation with critical path identification
3. **Risk Engine** (`server/src/engine/riskEngine.ts`) — Five-dimension risk scoring with evidence and mitigations

Results are displayed across three panels:
- **CostBreakdownTable** — Department-by-department cost ranges with drivers
- **ScheduleCard** — Timeline with phases and risk flags
- **RiskScoreCards** — Security, legal, ESG, access, and data completeness scores

### Step 5 — Save to Branch

The user clicks **SaveToBranchButton** (`src/components/simulation/SaveToBranchButton.tsx`) to persist the simulation as an artifact on the current branch. This creates a SIMULATION-type artifact with the full inputs and outputs.

### Step 6 — Compare Scenarios (Optional)

The user runs a second simulation (e.g., with helicopter logistics and high security posture) and uses **ScenarioCompare** (`src/components/simulation/ScenarioCompare.tsx`) to see the delta. The compare engine (`server/src/engine/compareEngine.ts`) provides a structured recommendation.

### Step 7 — Open Project Builder

The user navigates to the **Project Builder** screen (`src/components/screens/ProjectBuilder.tsx`), which guides them through a multi-step wizard:

1. **ChooseGoalStep** (`src/components/project-builder/ChooseGoalStep.tsx`) — Select project type and objectives
2. **SelectAreaStep** (`src/components/project-builder/SelectAreaStep.tsx`) — Define the target geographic area
3. **AssumptionsStep** (`src/components/project-builder/AssumptionsStep.tsx`) — Configure operational assumptions
4. **GeneratePlanStep** (`src/components/project-builder/GeneratePlanStep.tsx`) — Generate the plan (calls `POST /api/repos/:id/branches/:id/project-plan/generate`)

### Step 8 — Review Generated Plan

The builder service (`server/src/services/projectBuilder.service.ts`) produces:
- **Task Groups** — Phased tasks with duration estimates (min/P50/max) per department
- **Document Checklists** (`server/src/templates/docChecklists.ts`) — Regulatory and operational document requirements
- **Risk Register** — Pre-populated risks with likelihood, impact, and mitigations
- **Budget Summary** — Linked to the simulation's P50 cost data (if simulation_id provided)
- **Timeline Summary** — Aggregated phase durations

These are displayed in the **GeneratedTaskBoard** (`src/components/project-builder/GeneratedTaskBoard.tsx`) and **DocsChecklist** (`src/components/project-builder/DocsChecklist.tsx`).

### Step 9 — Export and Commit

The **ExportCommitStep** (`src/components/project-builder/ExportCommitStep.tsx`) allows the user to commit the generated plan as a PLAN-type artifact on the branch, creating a permanent record with SHA-256 provenance.
